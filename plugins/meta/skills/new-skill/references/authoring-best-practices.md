# Skill authoring — house best practices

House checklist for writing or editing a skill in this marketplace. The deep, tested methodology
lives in **`superpowers:writing-skills`** — read it for anything non-trivial. This file is the
house-specific layer + a quick checklist; it does not repeat superpowers.

## The rules that matter most

- **`description` = WHEN, not WHAT.** State the triggering conditions only (ideally opening "Use
  when…"). Never summarize the workflow in the description — an agent that reads the description
  acts on it and skips the body (a tested failure mode). Bad: "…Splits all changes into atomic
  commits." Good: "Use when the user asks to commit or push."
- **Body short; depth in `references/`.** Keep SKILL.md an overview + decision rules + pointers.
  Push long code, tables, walkthroughs into `references/<file>.md` and link one level deep — they
  load only when the reader follows the link, so the always-loaded body stays cheap.
- **One excellent example beats five mediocre ones.** Show the canonical case in full; don't
  enumerate variants.
- **Match the form to the failure.** If agents cut a corner under pressure, add a prohibition + a
  "thought → reality" rationalization table that intercepts the specific excuse. Otherwise give a
  positive recipe. (See `meta:ockham`, `meta:solid`, `salesforce:apex_test-authoring` for the table
  pattern in use.)
- **Keep the enforcement teeth inline.** MUST/NEVER lines, checklists, and ritual/persona machinery
  that changes behavior stay in the body — only relocate motivation and detail.

## House specifics (this marketplace)

- **No `name:` frontmatter.** The folder name is authoritative; the callable id is `<domain>:<folder>`
  (colon form is invalid in the field). Frontmatter is just `description:`.
- **`plugin.json`** (per domain) carries `name`, `description`, `version` (semver), `keywords`, `author`.
- **`marketplace.json` is generated by `sync.sh`** — never hand-edit it. `README.md` is hand-maintained.
- **English only** in every artifact.
- Full contributor contract → repo-root `AGENTS.md`.

## Before finalizing

- Description trimmed to triggering conditions (no workflow summary).
- Body ≤ ~1000 words, or the excess is justified enforcement/ritual; detail relocated to `references/`.
- If you changed a `description`, run it through `skill-creator`'s eval / `improve_description`
  before merge (triggering is load-bearing).
- Add the skill's one-line entry to `README.md` by hand.
